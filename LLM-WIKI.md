# LLM-WIKI — Claude Code 바이브 코딩 암묵지

> v-view 프로젝트에서 Claude Code(AI Agent)와 함께 작업하며 직접 체득한 노하우 모음  
> 교과서에 없는 실전 경험 위주로 기록

---

## 1. 프롬프트 설계 원칙

### 1.1 "범위 명시" 없이 요청하면 범위가 터진다

```
❌ "리포트 화면 개선해줘"
✅ "ReportScreen의 AppBar에 복사 버튼 하나만 추가해줘. 
    다른 파일은 건드리지 말 것."
```

**왜**: AI는 친절하게 "함께 개선할 만한 것"까지 수정한다.  
범위를 명시하지 않으면 열 파일을 건드린다.

---

### 1.2 "성공 기준"을 함께 제시해야 검증이 된다

```
❌ "시선 분산 버그 고쳐줘"
✅ "시선 분산이 연속 1초 미만이면 카운트되지 않는다는 걸
    테스트 케이스로 먼저 작성하고, 통과시켜줘"
```

**왜**: 기준 없는 버그 수정은 "고쳐진 것처럼 보이는" 코드를 만든다.

---

### 1.3 복잡한 요청은 단계별로 분리한다

```
❌ "아키텍처 설계 + 코드 스캐폴드 + 테스트까지 다 해줘"
✅ STEP 1: "아키텍처 설계 옵션 2개만 제시해줘"
   STEP 2: "1번으로 결정. 폴더 구조만 만들어줘"
   STEP 3: "이 폴더 구조로 도메인 엔티티 5개 작성해줘"
```

**왜**: 한 번에 큰 요청을 하면 AI가 가정을 많이 세우고 틀리는 부분이 생긴다.  
단계를 나누면 각 단계에서 직접 검토하고 방향을 수정할 수 있다.

## 2. 코드 생성 패턴

### 2.1 레이어 위반을 방지하는 프롬프트 패턴

v-view의 4레이어 구조(`ui → state → domain ← data`)에서  
AI는 습관적으로 `ui`에서 `data`를 직접 임포트하려 한다.

**해결**: CLAUDE.md에 레이어 규칙을 명시하고 모든 요청에서 이를 참조하게 한다.

```
"CLAUDE.md의 레이어 규칙을 지켜줘. ui에서 data를 직접 참조하지 말 것."
```

---

### 2.2 Riverpod StateNotifier 패턴 — AI가 자주 틀리는 부분

```dart
// AI가 자주 쓰는 잘못된 패턴 (state를 직접 mutation)
state.someList.add(item); // ❌

// 올바른 패턴 (새 객체로 교체)
state = state.copyWith(someList: [...state.someList, item]); // ✅
```

**왜**: Riverpod는 state 객체 자체가 바뀌어야 rebuild가 트리거된다.

---

### 2.3 Hive — AI가 `TypeAdapter` 자동 생성을 권유한다

AI는 Hive를 쓸 때 `@HiveType`, `@HiveField`, `build_runner`를 권유한다.  
v-view에서는 수동 Map 직렬화를 선택했다:

```dart
// AI 추천 (자동 생성 — 7주 일정에 build_runner 설정 시간 낭비)
@HiveType(typeId: 0)
class SessionReport extends HiveObject { ... }

// v-view 선택 (Map 직렬화 — 설정 없이 바로 동작)
box.put(id, {'gazeRate': metrics.gazeRate, ...});
```

**학습**: 도구의 "권장 방식"이 항상 옳지 않다.  
일정과 학습 목표에 맞는 방식을 직접 선택해야 한다.

## 3. AI와 협업하는 법

### 3.1 "검토" 단계를 생략하지 않는다

AI가 생성한 코드는 반드시 직접 읽어야 한다.  
"동작한다"는 것이 "올바르다"를 의미하지 않는다.

v-view에서 실제로 발견한 케이스:
- `GazeAnalyzer.computeMetrics()`가 Data 레이어에 있었음 → Domain으로 이동
- `GazeFrame` 클래스가 Data 파일 안에 정의되어 있었음 → Domain 엔티티로 분리
- `gazeAnalyzerProvider`가 Data 객체를 반환하면서 State 파일에 있었음

**교훈**: AI는 "기능적으로 동작하는" 코드를 만들지만,  
아키텍처 일관성은 사람이 검토해야 한다.

---

### 3.2 CLAUDE.md = AI 팀원을 위한 온보딩 문서

CLAUDE.md를 잘 작성하면 매 프롬프트에서 같은 설명을 반복하지 않아도 된다.  
v-view의 핵심 규칙들:

```
- 시선 분산 기준: 연속 1초 이상
- 응시율 공식: (응시 프레임 / 전체 프레임) × 100
- API 키 하드코딩 금지
- MVP 범위 이탈 금지
```

이를 CLAUDE.md에 넣으면 AI가 세션마다 자동으로 참조한다.

---

### 3.3 "이미 구현된 건지" 먼저 확인한다

AI는 이미 존재하는 코드를 모르고 중복으로 만들 수 있다.  
매 작업 전: "이 기능이 이미 있는지 확인해줘"를 습관화한다.

v-view 사례: 재시도 버튼(TODO 1.3)이 이미 `ErrorDisplay` 위젯에 구현되어 있었다.  
확인 없이 진행했다면 중복 구현 + 충돌이 발생했을 것.

## 4. 생산성 팁

### 4.1 Git 브랜치 전략 — 주차별 분리

```
main (항상 동작하는 버전)
  └── week1 (오리엔테이션)
  └── week2 (구현 1)
  └── week3 (구현 2 + 문서)
```

주차별로 브랜치를 만들면 발표 시 "이 주에 무엇을 했는가"가 명확해진다.

---

### 4.2 오류 메시지를 통째로 붙여넣는다

```
❌ "빌드 오류가 있어요, 고쳐줘"
✅ [오류 메시지 전체를 복사해서 붙여넣기]
```

**왜**: AI는 오류 메시지의 파일명, 줄 번호, 코드를 모두 참조해서 정확히 수정한다.

---

### 4.3 `flutter analyze` → `flutter test` 순서를 지킨다

코드 작성 후 항상 이 순서:
1. `flutter analyze` — 타입 오류, 미사용 임포트 확인
2. `flutter test` — 로직 검증
3. 기기 실행 — UI/UX 확인

AI가 생성한 코드도 예외 없이 이 순서를 따른다.

## 5. 한계와 대응

| AI의 한계 | 대응 방법 |
|-----------|-----------|
| 실기기 동작 보장 불가 | 에뮬레이터 → 실기기 순서로 테스트 |
| 카메라/센서 연동 오류 빈번 | 기기별 퍼미션·해상도 직접 확인 |
| API 비용 감각 없음 | 세션당 호출 횟수 제한 직접 설계 |
| 개인정보 정책 판단 불가 | 원본 영상 저장 금지 규칙 직접 명시 |
| 한국 법규/용어 부정확 | 법률 관련 문구는 직접 검토 |

## 6. Claude Code 특화 기능

### 6.1 CLAUDE.md 자동 로드

프로젝트 루트의 `CLAUDE.md`는 매 세션마다 자동으로 컨텍스트에 포함된다.  
여기에 프로젝트 규칙, 레이어 구조, 금지 사항을 넣으면  
매번 설명하지 않아도 된다.

### 6.2 병렬 파일 읽기

Claude Code는 여러 파일을 동시에 읽을 수 있다.  
"이 3개 파일을 보고 아키텍처 일관성을 검토해줘"처럼 요청하면  
파일들을 병렬로 읽고 비교 분석한다.

### 6.3 메모리 시스템

`~/.claude/projects/` 폴더에 대화 간 기억을 유지한다.  
"앞으로 커밋은 요청할 때만 해줘" 같은 지시를 한 번 하면 다음 세션에도 유지된다.
