## 1. Think Before Coding — 코딩 전에 먼저 생각하기

- 요청이 모호하면 **반드시 질문**하고 코딩을 시작하지 마라.
- 구현 방식이 2가지 이상이면 **장단점을 제시**하고 선택하게 하라.
- 기능명세서에 명시된 내용과 다른 방향이면 **먼저 확인**하라.

```
❌ 그냥 구현 시작
✅ "시선 분산 감지를 ML Kit로 할지 MediaPipe로 할지 
    각각 장단점이 있습니다. 어떤 걸 선택하시겠어요?"
```

---

## 2. Simplicity First — 단순함이 최우선

- MVP 기능만 구현하라. **v2 기능(STT, 표정분석)은 절대 먼저 짜지 마라.**
- 요청하지 않은 추상화, 유틸 클래스, 헬퍼 함수를 만들지 마라.
- 동작하는 최소한의 코드가 복잡한 완벽한 코드보다 낫다.
- 일어날 수 없는 시나리오에 대한 오류 처리를 추가하지 마라.
- 200줄로 짠 코드가 50줄로 가능하다면 다시 써라.
  스스로에게 물어라: "시니어 엔지니어가 이걸 과복잡하다고 할까?" → Yes면 줄여라.

```
MVP 범위:
✅ 시선 분석 (ML Kit)
✅ AI 질문 생성 (OpenAI gpt-4o-mini)
✅ 세션 피드백 리포트
✅ 로컬 히스토리 (Hive)
✅ 타이머/연속 질문

❌ STT → v2
❌ 표정/감정 분석 → v2
❌ 공유 기능 → v2
```

## 3. Surgical Changes — 수술하듯 정확하게 수정하기

- 요청한 부분**만** 수정하라. 나머지 코드는 건드리지 마라.
- 기존 주석, 변수명, 코드 스타일을 임의로 바꾸지 마라.
- 파일 전체를 다시 쓰지 말고 **변경된 부분만** 보여줘라.
- 인접한 "개선할 것 같은" 코드를 건드리지 마라. 발견하면 언급만 하라.
- 내 변경이 만들어낸 orphan(미사용 import·변수·함수)은 **내가 직접 정리**한다.
  단, 기존에 있던 dead code는 요청받기 전까지 삭제하지 않는다.

**판단 기준**: 변경된 모든 줄이 사용자의 요청에서 직접 추적 가능해야 한다.

```
❌ 파일 전체 재작성
✅ "_calculateGazeRate() 함수만 수정했습니다"
```

## 4. Goal-Driven Execution — 목표 중심으로 실행하기

- "고쳐줘" 대신 **검증 가능한 성공 기준**을 제시하라.
- 구현 전에 **테스트 케이스**를 먼저 작성하라.
- 완료 기준을 명확히 하고, 그 기준을 통과했는지 확인하라.
- 단계가 여러 개인 작업은 시작 전 계획을 먼저 제시하라:

```
1. [단계] → 검증: [확인 방법]
2. [단계] → 검증: [확인 방법]
3. [단계] → 검증: [확인 방법]
```

명확한 성공 기준이 있으면 독립적으로 반복 실행 가능하다.  
"되게 해줘" 같은 약한 기준은 매번 재확인이 필요하다.

```
❌ "시선 분석 버그 고쳐줘"
✅ "분산 상태가 연속 1초 이상일 때만 횟수가 카운트되는지
    검증하는 테스트를 먼저 짜고, 통과시켜줘"
```

## 5. v-view 프로젝트 컨텍스트

### 기술 스택
```
Framework  : Flutter (Android / iOS)
State      : Riverpod
Vision     : ML Kit face detection
AI         : OpenAI API (gpt-4o-mini)
Local DB   : Hive
Network    : dio
Env        : flutter_dotenv
Deploy     : Firebase App Distribution
```

### 레이어 구조
```
lib/
├── ui/          # 화면 및 위젯
├── state/       # Riverpod providers
├── domain/      # Use cases
└── data/
    ├── local/   # Hive
    └── remote/  # OpenAI API, ML Kit
```

### 핵심 규칙
- API 키는 절대 하드코딩 금지 → `.env` 파일 사용
- 원본 카메라 영상/오디오 로컬 저장 금지
- 시선 분산 카운트 기준: **연속 1초 이상** 유지 시 1회
- 응시율 공식: `(응시 프레임 수 / 전체 측정 프레임 수) × 100`
- 측정 품질 낮을 때: `'측정 불가/참고용'` 표시

### 오류 처리 원칙
- 네트워크 오류 → 재시도 버튼 + 원인 안내
- AI 요청 실패 → 최소 리포트(시선 지표만) 대체 제공
- 카메라 권한 거부 → 설정 이동 안내
- 백그라운드 전환 → 타이머 자동 일시정지

## 6. 디자인 시스템

- 메인 컬러 `#00C9A7` (민트-틸), 보조 컬러 `#FF6B6B` (코랄)
- 배경 흰색, 텍스트 `#1A2238`, 모서리 반경 16~20px, 라이트모드 전용
- 친근하고 모티베이션을 주는 톤 (Duolingo 스타일)
- **규칙**: 모든 화면은 `lib/theme`의 테마와 `lib/widgets`의 공통 위젯을 재사용할 것
- **버튼 규칙**: 메인 CTA 버튼은 반드시 `VViewButton` 또는 테마가 적용된 `ElevatedButton`만 사용
  `FilledButton` 사용 금지 — `primary(#006B58)`가 적용되어 `primaryContainer(#00C9A7)` 기반 디자인과 색상이 어긋남

## 7. 커밋 메시지 규칙

```
feat: 새 기능 추가
fix: 버그 수정
docs: 문서만 수정
refactor: 기능 변경 없는 코드 개선
test: 테스트 추가/수정
chore: 빌드/설정 변경
```

예시:
```
feat: OpenAI gpt-4o-mini 기반 AI 질문 생성 구현 (기능명세서 1.1)
fix: 시선 분산 횟수 1초 미만 흔들림 제외 처리
test: 응시율 계산 로직 유닛 테스트 추가
```
