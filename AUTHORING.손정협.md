# AUTHORING.손정협.md
## AI Agent 부트스트래핑 방법론 — v-view 프로젝트

> 이 파일 하나로 새 저장소에서 Claude Code가
> 프로젝트의 모든 Agent 설정을 자동 생성할 수 있다.

---

## 1. 이 파일의 목적

프로젝트를 처음 시작하거나 팀원·다른 기기에서 이어받을 때,
Claude Code에 이 파일을 제공하면:

- `AGENTS.md` — 저장소 전역 Agent 지침 재생성
- `md/CLAUDE.md` — 코딩 원칙 재생성
- `docs/ARCHITECTURE.md` — 아키텍처 문서 재생성
- `docs/ADR/` — 결정 기록 재생성
- `docs/SETUP.md` / `DEPLOY.md` / `TESTING.md` — 운영 문서 재생성

---

## 2. 부트스트래핑 방법

Claude Code에 아래 명령을 입력하면 된다:

```
AUTHORING.손정협.md 파일을 읽고,
섹션 4~8의 명세를 기반으로 해당 파일들을 생성해줘.
기존 파일이 있으면 내용을 비교하고 차이점만 알려줘.
```

---

## 3. 나의 Vibe Coding 워크플로우

### 3.1 세션 시작 루틴

```
1. Claude Code 실행
2. "AGENTS.md와 md/CLAUDE.md를 읽어줘" → 컨텍스트 로드
3. 오늘 작업할 기능 번호 선언: "기능명세서 1.2 꼬리 질문 생성을 구현해줘"
4. Claude가 구현 후 → 내가 코드 리뷰 (이해 못하면 설명 요청)
5. flutter analyze 통과 확인
6. 내 말로 구현 내용 설명 가능한지 자기 점검
```

### 3.2 모르는 코드가 나왔을 때

```
"이 함수가 정확히 무엇을 하는지 한 줄로 설명해줘"
"이 패턴을 쓴 이유가 뭔지 다른 대안과 비교해서 설명해줘"
"면접에서 이 코드에 대해 질문받는다면 어떻게 답해야 해?"
```

**원칙**: 설명 못하면 그 코드는 쓰지 않는다.

### 3.3 커밋 전 자기 점검 체크리스트

- [ ] 이 변경의 목적을 한 문장으로 말할 수 있다
- [ ] 변경된 파일의 레이어 역할을 설명할 수 있다
- [ ] 시선 분산 1초 기준이 깨지지 않았다 (`flutter test` 확인)
- [ ] API 키가 코드에 없다
- [ ] `flutter analyze` 이슈 없음

## 7. ADR 생성 명세

각 ADR은 다음 형식을 따른다:

```markdown
# ADR-00X: [결정 제목]

## 상태
Accepted / Deprecated / Superseded by ADR-00Y

## 컨텍스트
[왜 이 결정이 필요했는지]

## 결정
[무엇을 선택했는지 한 문장]

## 이유
[번호 목록, 구체적 근거]

## 결과
[실제 영향 및 트레이드오프]

## 대안 검토
[기각한 대안들과 이유 테이블]
```

v-view 필수 ADR 목록:
- ADR-001: 크로스플랫폼 프레임워크 선택 (Flutter)
- ADR-002: 상태 관리 선택 (Riverpod)
- ADR-003: 로컬 저장소 선택 (Hive)
- ADR-004: AI 백엔드 선택 (OpenAI gpt-4o-mini)

## 9. 이 방법론을 만든 이유

Vibe Coding에서 AI가 코드를 만들어주면,
**"내가 만든 것"이 아니라 "AI가 만든 것"** 이 되기 쉽다.

이를 방지하기 위해:
1. **문서 우선**: 코드 전에 명세를 만들면 내가 설계를 이해하게 된다
2. **명세 파일**: AUTHORING.md가 있으면 "이 코드가 왜 이렇게 됐는지"를 항상 설명할 수 있다
3. **자기 점검**: 커밋 전 체크리스트로 이해하지 못한 코드를 걸러낸다

> "AI가 만든 것을 내가 완전히 이해하고 설명할 수 있는가"가
> 이 방법론의 유일한 성공 기준이다.

## 10. 라이브 부트스트래핑 시연 스크립트 (약 5분)

> 발표 당일 새 터미널에서 실시간으로 기능을 추가하는 시연

### 준비물

- Claude Code 실행 가능한 노트북
- v-view 저장소 클론 완료 + `.env` 설정 완료
- 에뮬레이터 또는 실기기 연결

### 시연 흐름

**[0:00] 컨텍스트 주입**
```
AGENTS.md와 md/CLAUDE.md를 읽어줘.
오늘 추가할 기능: [선택한 기능명]
```

**[0:30] 기능 요청 — 성공 기준 포함**
```
[기능명]을 구현해줘.
성공 기준:
1. [검증 방법 1]
2. [검증 방법 2]
CLAUDE.md의 레이어 규칙 준수, 다른 파일은 건드리지 말 것.
```

**[2:00] 코드 리뷰 포인트 (발표 중 설명)**
- "이 함수가 왜 domain 레이어에 있나요?" — 비즈니스 로직 = domain 규칙
- "State에서 왜 copyWith를 쓰나요?" — Riverpod rebuild 트리거 원리

**[3:30] 검증**
```bash
flutter analyze   # 이슈 0 확인
flutter test      # 테스트 통과 확인
```

**[4:30] 커밋**
```bash
git add [수정된 파일만]
git commit -m "feat: [기능명] 실시간 구현 시연"
```

### 시연 기능 후보 (난이도 낮은 순)

| 기능 | 예상 소요 | 설명 가능 포인트 |
|------|-----------|------------------|
| 리포트 날짜 표시 추가 | 1분 | UI → domain entity 참조 |
| 면접 유형별 아이콘 | 2분 | switch 표현식, UI 레이어 |
| 히스토리 정렬 (최신순) | 3분 | State → Hive 데이터 흐름 |

### 시연 중 막혔을 때 대처

- **오류 발생 시**: `flutter analyze` 출력을 통째로 Claude에 붙여넣기
- **이해 안 되는 코드**: "이 줄이 왜 필요한지 한 문장으로 설명해줘"
- **시간 초과 시**: 구현 완료 후 "다음 단계는 X" 라고 정리하고 마무리
